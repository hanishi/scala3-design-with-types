# Step 3: 変位の実践

## 3-1. 共変の実践 ― すでに使っている

共変を見るためにカスタム型は必要ありません ― 毎日使う標準ライブラリの型に入っています。
どれも 2-5 で見た `[B >: A]` 下限境界の脱出口を示しています：

```scala
{{#include ../../../examples/step2/step2h.scala}}
```

このコードは `Nothing` ― Scala の**ボトム型**を紹介します。すべての型のサブタイプであり、
`Nothing` 型の値は決して存在できません。

コンパイラは型の片側が未指定のときにプレースホルダーとして使います：

- `Right(book)` は `Right[Nothing, Book]` ― Left 型はまだない
- `Left("error")` は `Left[String, Nothing]` ― Right 型はまだない
- `List()` は `List[Nothing]` ― 空、要素型はまだない
- `None` は `Option[Nothing]` ― 値がない、型もまだない

共変がこれを機能させます。任意の `A` に対して `Nothing <: A` なので、
型が期待される場所にいつでも適合します。2-5 ですでに見ました：`Cart()` は
`Cart[Nothing]` です。`Book` を追加すると `[B >: A]` によって `Cart[Book]` に拡大します。

パターンはすべて同じです：型は `A` の値を**生産**（返す、保持する、発行する）するので、
`+A` が自然です。異なるサブタイプの値を追加・結合する必要があるときは、
`[B >: A]` が安全に型を拡大します。

## 3-3. では不変は本当に役立つのか？

はい、ただし役割はあなたが思うより狭いです。

不変は型が**読み書き両方する**ときに正しい選択です ― 可変コンテナ、
双方向チャネル、読み書き参照。しかし良く設計された Scala コードでは、
不変性が好まれるため比較的稀です。

```scala
{{#include ../../../examples/step2/step2j.scala}}
```

**実際のところ：**

Scala で設計する型のほとんどは二つの陣営のどちらかに入ります：

```
A を生産する（読み取り専用データ、結果、イベント、ストリーム） → 共変にする [+A]
A を消費する（ハンドラ、ライター、バリデータ、順序付け）       → 反変にする [-A]
```

不変は**アノテーションしない**ときに得られるものです ― そしてそれは多くの場合、
自分の型がプロデューサーなのかコンシューマーなのかまだ考えていないからです。
考えてみると、`+` か `-` のどちらかが大抵当てはまることに気づくでしょう。

「この型は `A` を生産するのか消費するのか？」と問う規律自体が
貴重な設計エクササイズです。型の役割を明確にすることを強制し、
コンパイラがその答えを検証してくれます。
